# STEP 5 振り返りと次のステップ

## この章で学ぶこと

このチュートリアルも最後のステップです。

**この章では、ここまで学んだことを振り返り、自分の業務への応用を考えます。** Agent Teams の使い方を「知っている」から「使える」に変えるには、自分の文脈に引き寄せて考えることが有効です。

## このチュートリアルで体験したこと

5 つのステップを通じて、以下を体験しました。

STEP 1 では、Agent Teams が何者かを理解しました。複数の Claude Code インスタンスがチームとして協調動作する仕組みであること、subagents との違い、向いているケースと向かないケースを学びました。

STEP 2 では、Claude Code をインストールし、Agent Teams を有効化する環境を整えました。`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` という環境変数が必要なことと、v2.1.32 以降が必要なことを確認しました。

STEP 3 では、実際に Agent Teams を動かしました。1 つのコードに対して「可読性」「パフォーマンス」「セキュリティ」の 3 つの観点から並列にレビューさせ、各エージェントが独立した視点で動いている様子を見ました。

STEP 4 では、プロンプトの書き方を工夫しました。役割を「何を見るか」で定義すること、出力フォーマットを統一すること、担当外への踏み込みを制限すること、進行と終了条件を明示することで、レビューの質と扱いやすさが変わることを学びました。

## まとめ

Agent Teams は、使い方によっては明確な価値があります。ただし「使えば必ず良くなる」というものでもありません。

**並列化が本当に意味を持つとき**、つまり「同じ対象を異なる視点で独立して見てほしい」という状況には、Agent Teams はよく合います。コードレビューはその典型ですが、仕様書のレビューや、複数の設計案の評価なども同じ性質を持ちます。

**プロンプトの設計に手間をかけられるとき**、Agent Teams は価値を発揮します。指示が曖昧だとメンバーが同じことを繰り返し指摘するだけで、単一エージェントに聞くのと変わらない結果になります。

**時間とトークンに余裕があるとき**が前提となります。複数エージェントが並列で動く分、調整のオーバーヘッドとトークン消費は増えます。小さな作業を手軽に試したいだけなら、通常の Claude Code の方が早いこともあります。

## 自分の業務への応用を考える

コードレビュー以外にも、「複数の独立した視点」が価値を持つ場面はあります。

**技術文書や仕様書のレビュー**に使えるかもしれません。「はじめてこの機能に触れる開発者」「実装を担当するバックエンドエンジニア」「インフラの影響を気にする SRE」という 3 つの立場から、API 仕様書を並列でレビューさせる、という使い方です。

**設計の多角的な検証**にも向いています。データベーススキーマの設計案を「拡張性」「クエリ効率」「マイグレーションの容易さ」という観点から並列で評価させる、といった用途が考えられます。

**想定ユーザーの反応シミュレーション**も可能性があります。新機能の UX を「初心者ユーザー」「パワーユーザー」「アクセシビリティに配慮が必要なユーザー」の視点から並列で評価させる、という使い方も考えられます。

ただし、これらはどれも「並列化したときに独立した視点が出せるかどうか」がカギになります。担当を分けても、それぞれの視点が似たり寄ったりの内容になってしまう場合は、Agent Teams を使う意味が薄れます。試す前に「視点が本当に分かれるか」を考えておくと、ムダなトークン消費を避けられます。

## これから試すこと

まず、**自分のコードでやってみます**。STEP 3 で使ったサンプルコードではなく、自分が今取り組んでいる実際のコードでレビューを試してみると、サンプルコードでは気づかなかった発見があります。

次に、**プロンプトは繰り返し書き直すことが前提**です。STEP 4 のテクニックは出発点に過ぎません。自分の用途に合った観点の設定、フォーマット、制約のバランスは、試しながら見つけていくものです。

最後に、**単一エージェントという選択肢も常に持っておきます**。Agent Teams は強力ですが、単一エージェントで十分なケースも多いです。手間とコストに見合う場面でだけ使う、というメリハリが長く使い続けるコツです。

## 実務投入前チェックリスト

自分の業務で Agent Teams を使う前に、次の項目を確認します。

- 並列化する価値がある作業か
- teammate ごとの担当観点や担当範囲が明確か
- 複数の teammate が同じファイルを編集する可能性は低いか
- 成功条件や最終アウトプットの形式を指定しているか
- 単一エージェントより時間やトークンを使ってもよい作業か

このチェックリストでいくつも引っかかる場合は、Agent Teams ではなく単一の Claude Code セッションや subagents で十分かもしれません。逆に、担当観点が明確に分かれ、互いに独立して調査できるなら、Agent Teams を試す価値があります。

## チュートリアルを終えて

このチュートリアルを通じて、Agent Teams の仕組みと使い方の基礎を学びました。

実験的機能ゆえの制限もありますし、プロンプト設計には試行錯誤が要ります。すぐに魔法のように動くとは限りません。ただ、複数の独立した視点を並列で走らせるという考え方は、うまく使えば確かに価値があります。

自分のプロジェクトに当てはめると、合う場面が見えてくるのではないでしょうか。
