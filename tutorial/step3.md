# STEP 3 はじめての Agent Teams：コードレビュー

## この章で学ぶこと

環境が整ったので、実際に Agent Teams を動かします。

**この章では、1 つのコードに対して複数の観点から並列にレビューさせます。** 「可読性」「パフォーマンス」「セキュリティ」の 3 つの観点を担当するエージェントが同時に動き、それぞれ独立したレビューを返す様子を見ていきます。

## レビューするコードを用意する

まず、レビュー対象のコードを用意します。ここでは、ユーザーの注文データを取得して合計金額を計算する Python 関数を使います。いくつかの問題を含んだコードです。

作業用ディレクトリを作って、ファイルを置きます。

```bash
mkdir agent-teams-demo
cd agent-teams-demo
```

`orders.py` というファイルを作り、以下の内容を書きます。
このコードは、チュートリアル用に作成した架空のサンプルです。

```python
def process_orders(user_id, discount=0):
    import database
    conn = database.connect()
    orders = conn.query("SELECT * FROM orders WHERE user_id = " + str(user_id))
    total = 0
    result = []
    for order in orders:
        price = order['price']
        if discount > 0:
            price = price - (price * discount / 100)
        total = total + price
        result.append({
            'id': order['id'],
            'price': price,
            'status': order['status']
        })
    conn.close()
    return {'orders': result, 'total': total}
```

このコードには意図的にいくつかの問題を仕込んであります。問題点はエージェントのレビュー後に照合できます。

## Agent Teams を起動して、プロンプトを送る

`agent-teams-demo` ディレクトリで Claude Code を起動します。

```bash
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 claude
```

Claude Code のプロンプトが表示されたら、以下のプロンプトをそのまま貼り付けて送信します。

```
orders.py のコードを、以下の3つの観点から並列にレビューしてください。
それぞれ独立したエージェントが担当するようにチームを組んでください。

1. 可読性エージェント：コードの読みやすさ、命名規則、構造の明確さを評価する
2. パフォーマンスエージェント：処理速度、データベースアクセスの効率、メモリ使用量を評価する
3. セキュリティエージェント：セキュリティ上のリスク、脆弱性、危険な実装パターンを評価する

各エージェントは他のエージェントの視点に引っ張られることなく、自分の担当観点だけに集中してください。
最後に、3つのレビューをまとめてください。
```

## 何が起きているのかを見る

プロンプトを送ると、Claude Code がしばらく考えたあとで動き始めます。ターミナルには、リーダーエージェントがチームを編成してタスクを分配している様子が表示されます。

リーダーエージェントはまずタスクリストを作成し、3 つのレビュータスクをそれぞれのメンバーエージェントに割り当てます。3 人のメンバーが同時に動き始め、それぞれが `orders.py` を独立した視点から分析します。

各メンバーが作業中、ターミナルには処理の進捗が流れます。メンバー同士はメールボックスを通じてやり取りもできるため、「セキュリティエージェントが見つけた SQL インジェクションの問題をパフォーマンスエージェントにも共有する」といった連携が起きることもあります。

3 つのレビューが出揃ったら、リーダーエージェントが結果を統合して最終的な回答をまとめます。

---

## チームが起動したことを確認する

Agent Teams が正しく起動している場合、単に Claude がひとりで回答するのではなく、teammate が作成されます。成功しているかは、次の点で確認できます。

- リーダーが複数の teammate を作成している
- 共有タスクリストに「可読性レビュー」「パフォーマンスレビュー」「セキュリティレビュー」のようなタスクが並ぶ
- 各 teammate がそれぞれの観点で `orders.py` を読んでいる
- 最後にリーダーが各レビューを統合している

in-process モードでは、すべての teammate が同じターミナル内で動きます。`Shift+Down` を押すと、リーダーと teammate の表示を切り替えられます。`Ctrl+T` を押すと共有タスクリストを確認できます。

split panes モードでは、tmux や iTerm2 の pane に teammate ごとのセッションが表示されます。各 pane を見れば、どの teammate が何を担当しているか確認できます。

もし teammate が作成されず、Claude が通常の単一回答として 3 観点のレビューをまとめて返してきた場合は、Agent Teams のハンズオンとしては成功していません。その場合は、次のようにより明示的に依頼します。

```
Agent Team を作成してください。
3人の teammate を作り、それぞれに可読性、パフォーマンス、セキュリティのレビューを割り当ててください。
全 teammate のレビューが完了するまで待ってから、リーダーが結果を統合してください。
```

## 結果を読む

レビューの結果には、それぞれの観点ならではの指摘が入ります。

**可読性の観点**からは、関数の中に `import database` が入っていることや、変数名の曖昧さ、エラーハンドリングがないことなどが指摘されます。

**パフォーマンスの観点**からは、`SELECT *` で全カラムを取得していること、大量の注文データに対するページングや集計処理がないこと、コネクション管理が単純すぎることなどが挙がります。

**セキュリティの観点**からは、文字列結合による SQL クエリの組み立て（SQL インジェクションの典型的なパターン）が真っ先に指摘されます。割引率 `discount` の入力検証がないことや、例外が起きたときに接続が閉じられない可能性も指摘されるかもしれません。

3 つを並べて読むと、同じコードに対して異なる「目」が当たっていることが確認できます。可読性エージェントが見落とすセキュリティリスクを、セキュリティエージェントはしっかり捉えています。それが Agent Teams の使いどころです。

期待される指摘の例は、次のようなものです。

```text
セキュリティ:
- user_id を文字列結合して SQL を作っているため、SQL インジェクションの危険がある
- discount の範囲チェックがなく、不正な値を渡せてしまう

パフォーマンス:
- SELECT * で不要なカラムまで取得している
- 注文数が多い場合のページングや DB 側集計が考慮されていない

可読性:
- DB接続、クエリ、割引計算、レスポンス整形が1つの関数に混ざっている
- 例外時の処理がなく、正常系だけを前提にしている
```

完全に同じ文言になる必要はありません。重要なのは、各 teammate が担当観点ごとに異なる指摘を出していることです。

## 体験して分かること

実際に動かしてみると、いくつかのことがわかります。

まず、**3 つの視点が独立している**ことが確認できます。1 人のレビュアーが「パフォーマンスもセキュリティも可読性も」と一気に見るとき、どうしても得意な観点に引っ張られてしまいます。Agent Teams では、各エージェントが自分の担当観点に専念するため、均等な深さで見てもらえます。

次に、**プロンプトの品質が結果に直結する**ことも分かります。「3 つの観点でレビューして」という指示だけでは、エージェントそれぞれの役割が曖昧になりがちです。「可読性エージェントは〇〇を評価する」と具体的に書くほど、レビューの質が上がります。この点については STEP 4 で掘り下げます。

最後に、**小さなレビューでは単一セッションより処理時間が長く感じられることがあります**。複数エージェントが起動してタスクを調整する分のオーバーヘッドがあるためです。手軽に試したいだけなら単一エージェントに聞く方が早いこともあります。Agent Teams を使う価値があるのは、「並列化による独立した視点」を必要としているときです。
