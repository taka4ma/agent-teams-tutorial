# STEP 1 Agent Teams とは何か

## この章で学ぶこと

コードレビューを例に考えてみましょう。セキュリティ、パフォーマンス、可読性という3つの観点を1人のレビュアーが順番に確認していく方法と、それぞれを専任のエージェントが同時に見る方法では、どちらが効率的でしょうか。Claude CodeのAgent Teamsは、後者を実現する仕組みです。

**この章では、Claude Code の Agent Teams という機能が何者なのかを学びます。** 具体的には、どういう仕組みなのか、どんな場面で威力を発揮し、どんな場面では向かないのかを理解することが目標です。読み終わったころには、「あの場面なら使えそうだ」という具体的なイメージを持てます。

## Claude Code とは（簡単に）

Agent Teams の話に入る前に、Claude Code 自体についてひと言触れておきます。

Claude Code は Anthropic が提供する AI コーディングアシスタントで、ターミナルから使う CLI ツールです。単にコードを提案するだけでなく、プロジェクトのファイルを読み書きし、コマンドを実行し、複数のファイルにまたがる変更を一度に行えます。LLMを使ったチャットツールとの最大の違いは、実際のファイルシステムにアクセスできることにあります。

Claude Code 自体のインストール方法や基本的な使い方は、このチュートリアルのシリーズでは別のステップで扱います。ここでは「Claude Codeはターミナルで動くAIエージェントだ」という理解で先に進みましょう。

## Agent Teams とは

Agent Teams は、**複数の Claude Code インスタンスがチームとして協調動作する仕組み**です。

通常、Claude Code を起動したらそれは 1 つのセッションとして動きます。あなたが指示を出し、Claudeが応答し、またあなたが指示を出す、というループです。Agent Teamsはこの単一セッションを拡張し、複数の独立したClaude Codeのインスタンスを同時に走らせることができます。

実際にチームを作るときは、ユーザーが「チームを組んでください」「teammate を作ってください」のように明示的に依頼します。Claude がチーム化を提案する場合もありますが、その場合もユーザーが承認してから進みます。

チームには**リーダー**と**メンバー**がいます。リーダーはあなたが直接対話するメインセッションで、作業を計画し、メンバーに指示を出し、結果を統合する役割を担います。メンバーはリーダーが生成した別のClaude Codeインスタンスで、それぞれが独自のコンテキストウィンドウ（記憶領域）を持ち、割り当てられたタスクに集中します。

チーム全体は**共有タスクリスト**を通じて調整されます。リーダーがタスクを作成し、メンバーがそれを引き受けて処理します。タスクには依存関係も設定でき、あるタスクが完了しないと次のタスクに着手できない、という順序制約も表現できます。

また、メンバー同士は**メールボックス**と呼ばれるメッセージングシステムを通じて直接通信できます。あるメンバーが調査した結果を別のメンバーに送ることも、議論して合意を形成することも、仕組みとして用意されています。

### 実験的機能であることに注意

現時点（2026年4月）で、Agent Teams は**実験的機能**という位置づけです。デフォルトでは無効になっており、使いたい場合は明示的に有効化する必要があります。
有効化する手順は STEP 2 で説明します。
Agent Teams はまだ開発途上の機能であり、今後大きく変わる可能性があります。今回のチュートリアルでは、現時点での仕様に基づいて説明しますが、将来的には内容が古くなる可能性があることをご了承ください。

## subagents との違い

Claude Code には Agent Teams とよく似た「subagents」という仕組みもあります。混乱しやすいので、ここで整理しておきます。

subagents は、メインのセッション内からヘルパーエージェントを一時的に立ち上げる仕組みです。ヘルパーが作業を終えたらメインエージェントに結果を返し、それ以上のことはしません。あくまでメインエージェントが全体を管理し、ヘルパーたちは結果だけを報告します。

Agent Teams との本質的な違いは「メンバー同士が直接対話できるかどうか」にあります。subagentsではメインエージェントを介さなければ情報が渡りませんが、Agent Teamsではメンバー同士がリーダーを経由せずに直接メッセージを交わせます。

コードレビューを「セキュリティ」「パフォーマンス」「可読性」の観点で並列化する場合、各エージェントが他の観点に影響されず独立した判断を下せるのは、メンバーが独立した環境で動いているからです。

もう一つの違いはコストです。subagentsは結果だけをメインコンテキストに返すためトークン消費が比較的抑えられます。対してAgent Teamsでは、各メンバーが独自のコンテキストウィンドウを持ちながら動くため、アクティブなメンバーの数に比例してトークン消費が増えます。

## どんなときに向いているか

Agent Teams が力を発揮するのは、「**複数の視点や作業ストリームを並列で走らせることに本当の価値があるとき**」です。

**調査・レビューの並列化**が最もわかりやすい用途です。PRのコードレビューを例にとると、セキュリティ、パフォーマンス、テストカバレッジという3つの観点を1人のレビュアーが順番に見ていくより、それぞれを専任のエージェントが同時に見た方が漏れが少ないです。

**独立したモジュールの同時開発**も向いています。フロントエンド・バックエンド・テストがそれぞれ独立して進められる作業なら、3人のメンバーが互いに足を踏み合わずに並列で動けます。

**競合する仮説でのデバッグ**も有効な使い方です。バグの原因が絞れていないとき、単一エージェントはひとつの仮説を見つけたらそこで止まりやすいです。複数のメンバーにそれぞれ異なる仮説を担当させ、互いに反証し合わせることで、本当の原因に収束しやすくなります。

## 向かないケース

Agent Teams は万能ではありません。むしろ、使わない方が良い場面の方が多いです。

**タスクが順序依存している場合**は、単一セッションの方がシンプルです。ステップAが終わらないとステップBに入れない、という直列の作業では、チームを組んでも並列化できる余地がありません。調整のオーバーヘッドが加わる分、むしろ遅くなります。

**複数のメンバーが同じファイルを編集する可能性がある場合**は、上書き事故が起きます。作業の分割が「独立したファイルのセット」単位にできないなら、Agent Teamsは適していません。

**ルーティンなタスク、あるいは小さなタスク**は、単一セッションの方がコスト効率が良いです。各メンバーが独自のコンテキストウィンドウを持つので、メンバー数に比例してトークン消費が増えます。作業が単純で、並列化しても時間短縮の恩恵がほとんどない場合は、費用対効果が合いません。

## Agent Teams の応用アイデア

コードレビューに慣れてきたら、同じ発想を他の場面に応用できます。キャラクター設定を細かくすれば、たとえば「コスト重視の購買担当者」「技術的な詳細を気にするエンジニア」「経営目線で見る意思決定者」というキャラクターを設定して、自分の提案書を並列でレビューさせる、といった使い方が考えられます。

ただ、これはまだ仮説です。実際に試すと、プロンプトの設計次第で品質は大きく変わりますし、うまくキャラクターに沿ったレビューが得られない場合もあります。Agent Teamsは「使えばすごいことが起きる魔法のツール」ではなく、うまく使えば価値を引き出せるがそれなりに試行錯誤が必要な道具だ、という認識が正確です。
