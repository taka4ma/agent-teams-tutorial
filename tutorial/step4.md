# STEP 4 プロンプトを工夫する

## この章で学ぶこと

STEP 3 で Agent Teams の基本的な動きを体験しました。結果は出たものの、「もっとこういう形で返してほしい」「もう少し深く見てほしい」と感じることもあります。

**この章では、Agent Teams から価値を引き出すためのプロンプトの書き方を学びます。** 役割定義の具体化、出力フォーマットの統一、観点ごとの評価軸の設定、といったテクニックを紹介します。

## なぜプロンプトの書き方が重要なのか

通常の単一エージェントへの指示では、文脈から補完してくれる余地があります。エージェントが解釈で補ってくれるからです。

Agent Teams では挙動が変わります。各メンバーエージェントは独立したコンテキストを持つため、役割の定義があいまいだと「自分が何を見れば良いのか」の解釈がメンバーごとにばらつきます。リーダーが曖昧な指示を各メンバーに伝えると、メンバーは独自の判断で動いてしまい、結果として「3 人がほぼ同じことを指摘した」ということが起きやすくなります。

プロンプトを丁寧に書くほど、メンバーが担当観点に集中でき、並列化の恩恵が得られます。

## テクニック 1：役割を「何を見るか」で定義する

多くの場合、役割名だけを渡して終わりにしてしまいがちです。

```
セキュリティエージェント、パフォーマンスエージェント、可読性エージェントの
3 人でレビューしてください。
```

これでも動きますが、各エージェントが「セキュリティ担当とはどういう仕事か」を自分で解釈してしまいます。

代わりに、**何を具体的に確認するか**を書きます。

```
以下の担当を割り当ててチームを組んでください。

セキュリティエージェント：SQLインジェクション、認証・認可の欠如、
機密情報の露出、エラーメッセージでの情報漏洩を確認する。

パフォーマンスエージェント：不要なデータベースアクセス、
ループ内の重複処理、メモリを無駄に消費するデータ構造、
N+1 問題を確認する。

可読性エージェント：関数の単一責任、命名の明確さ、
コメントが必要な箇所、エラーハンドリングの存在を確認する。
```

確認項目を明示することで、メンバーが迷わずに集中できます。

## テクニック 2：出力フォーマットを統一する

各メンバーが好き勝手なフォーマットで回答すると、リーダーが結果をまとめるときに苦労します。最終的なレポートも読みにくくなります。

メンバーごとに出力フォーマットを指定しておくと、読み比べやすくなります。

```
各エージェントは以下のフォーマットで回答してください。

## [担当観点]レビュー

**深刻度の高い問題**
- （該当なしの場合は「なし」と書く）

**改善すべき点**
- 

**良い点**
- 

**修正提案コード（必要な場合のみ）**
- （必要であればコードブロックで示す）
```

フォーマットを揃えると、3 つのレビューを横並びで読んだときに比較しやすくなります。深刻度を明示させることで、優先順位もはっきりします。

## テクニック 3：観点をあえて制限する

「なんでも見ていい」という指示は、各メンバーが他の観点にも踏み込む原因になります。セキュリティエージェントが可読性の問題も指摘し始めると、可読性エージェントとの重複が増えて並列化の意味が薄れます。

各メンバーに**担当外は見ない**ことを明示するのが有効です。

```
各エージェントは自分の担当観点だけを評価してください。
他の観点の問題に気づいても、担当エージェントに任せてコメントしないでください。
```

ただし、これを厳しくしすぎると、複数の観点に関わる重要な問題（たとえばセキュリティとパフォーマンスの両方に影響する設計上の欠陥）が抜け落ちることもあります。「担当外でも致命的な問題は一言だけ触れる」という緩衝ルールを加えるのも手です。

## テクニック 4：まとめ役の指示を追加する

リーダーエージェントは、メンバーの結果を自動的にまとめてくれることが多いです。しかし、まとめ方の指示を明示的に書いておくと、より使いやすい形式で出力されます。

```
最後に、リーダーエージェントは以下をまとめてください。

1. 修正必須の問題リスト（深刻度順）
2. 推奨改善事項リスト（重要度順）
3. 各観点から見た総合評価（1〜5点）
```

## テクニック 5：進行と終了条件を明示する

Agent Teams では、リーダーが teammate の完了を待たずに途中でまとめ始めたり、作業後の片付けが曖昧になったりすることがあります。特に初めて使うときは、レビューの品質だけでなく、進め方そのものもプロンプトに含めると安心です。

```
全 teammate のレビューが完了するまで待ってから統合してください。
未完了の teammate がいる場合は、結果をまとめずに進捗を確認してください。
最後に、teammate が残っていないか確認できる状態にしてください。
```

この指示を入れておくと、リーダーが「今ある結果だけでまとめる」のではなく、チーム全体の完了を確認してから統合しやすくなります。Agent Teams は実験的機能なので、作業後の状態確認まで明示しておくと、teammate が残っているかどうかも判断しやすくなります。

## 改善版プロンプトの全文

ここまでのテクニックをまとめた、改善版のプロンプトを示します。STEP 3 のプロンプトと比べて、結果の違いを確認できます。

```
orders.py のコードを、以下の構成でチームレビューしてください。

---
【チーム構成】

セキュリティエージェント
確認項目：SQLインジェクション、認証・認可の欠如、機密情報の露出、
エラーメッセージでの情報漏洩
担当外の問題は原則としてコメントしない（致命的な問題は一言触れる）

パフォーマンスエージェント
確認項目：不要なデータベースアクセス、ループ内の重複処理、
N+1 問題、メモリ効率
担当外の問題は原則としてコメントしない（致命的な問題は一言触れる）

可読性エージェント
確認項目：関数の単一責任原則、命名の明確さ、エラーハンドリング、
import 文の配置
担当外の問題は原則としてコメントしない（致命的な問題は一言触れる）

---
【出力フォーマット（各エージェント共通）】

## [担当観点]レビュー

**深刻度の高い問題**
- （なければ「なし」）

**改善すべき点**
-

**良い点**
-

---
【リーダーエージェントのまとめ】

1. 修正必須の問題リスト（深刻度順）
2. 推奨改善事項リスト
3. 総合評価コメント

【進行条件】

全 teammate のレビューが完了するまで待ってから統合してください。
未完了の teammate がいる場合は、結果をまとめずに進捗を確認してください。
最後に、teammate が残っていないか確認できる状態にしてください。
```

## プロンプトを書くうえで気をつけること

プロンプトを精緻にすることにもデメリットはあります。

長すぎるプロンプトはリーダーエージェントが各メンバーに渡す指示量を増やし、トークン消費が増えます。また、制約を細かくしすぎると、メンバーエージェントが本来なら有益な観察をしているのに「担当外だから言わない」という状況が起きます。

プロンプトは最初から完璧を目指すより、「まず動かして、結果を見て調整する」という繰り返しで育てる方が現実的です。
